### ✅ 1-1. PNG: Oxipng による前処理の導入（v1.2.0 で実装済み）

**実装内容**:
- `oxipng` クレートを Rust ライブラリとして組み込み（バイナリバンドル不要）
- 3段階パイプライン: pngquant → oxipng → ECT
- 不要チャンクの削除、フィルター最適化、アルファ最適化

**現状**: `pngquant` → `ECT`（2段階）  
**提案**: `pngquant` → `oxipng` → `ECT`（3段階）

#### 概要

[Oxipng](https://github.com/shssoichiro/oxipng) は Rust 製の PNG 最適化ツール。不要なメタデータチャンクの削除、フィルター戦略の事前選別、インターレース最適化などを行う。

#### 期待効果

- 不要チャンク（tEXt, iCCP, gAMA 等）の除去で **数%** 削減
- ECT への入力を事前に最適化することで、最終出力がわずかに改善

#### 実装方針

```
入力 PNG
  → pngquant (--quality=60-95, 減色)
  → oxipng (-o max --strip all, チャンク削除+フィルター最適化)
  → ECT (-9 --strict, ロスレス再圧縮)
  → 出力 PNG
```

#### 実装手順

1. **ワークフロー追加**: `brew install oxipng`（macOS）、GitHub Releases から取得（Windows）
2. **lib.rs 修正**: `compress_png()` に oxipng ステップを追加
3. **バンドル設定**: `build.yml` で oxipng バイナリをコピー

#### 優先度: ⭐⭐ （中）

> 現状でも十分プロ仕様。得られる改善はごく小さいが、実装コストも低い。

### 1-3. PDF: 画像抽出→個別最適化→再構築

**現状**: Ghostscript で一括ダウンサンプリング + JPEG 再圧縮  
**提案**: PDF 内の画像を個別に抽出し、Jpegli/ECT で最適化後に PDF に戻す

#### アーキテクチャ

```
入力 PDF
  → pdfimages (画像抽出)
  → 各画像を個別に最適化:
      JPEG → cjpegli で再圧縮
      PNG  → pngquant + ECT
  → img2pdf / Ghostscript で PDF に再構築
  → 出力 PDF
```

#### 実装手順

1. **pdfimages のバンドル**: Poppler ユーティリティに含まれる
2. **画像判定ロジック**: 抽出画像の形式に応じて適切な圧縮器を選択
3. **再構築**: 最適化画像を元の PDF に再埋め込み

#### 優先度: ⭐ （低）

> 実装の複雑度が非常に高い。Ghostscript の一括処理でも十分な結果が得られるため、見送り推奨。

## Phase 2: 次世代フォーマット対応

### 2-1. WebP 変換対応

#### 概要

Google 開発の画像フォーマット。JPEG/PNG の両方の用途をカバーし、主要ブラウザすべてで対応済み。

#### ツール

- **cwebp** (libwebp に含まれる)
- Homebrew: `brew install webp`
- Windows: [libwebp releases](https://developers.google.com/speed/webp/download)

#### 実装

```bash
# Lossy (JPEG代替)
cwebp -q 80 input.jpg -o output.webp

# Lossless (PNG代替)
cwebp -lossless input.png -o output.webp
```

#### 優先度: ⭐⭐⭐ （高）

> Web 制作者にとっての需要が非常に高い。実装も容易。

---

### 2-2. AVIF 変換対応

#### 概要

AV1 ベースの画像フォーマット。**現時点で Web 利用可能な形式の中で最高の圧縮率**を誇る。

#### ツール

- **avifenc** (libavif に含まれる)
- Homebrew: `brew install libavif`
- Windows: [libavif releases](https://github.com/AOMediaCodec/libavif/releases)

#### 実装

```bash
avifenc --min 20 --max 30 -s 6 input.png output.avif
```

#### 優先度: ⭐⭐ （中）

> 圧縮率は最強だが、エンコード速度が遅い。設定 UI のバランスが課題。

---

### 2-3. JPEG XL 変換対応

#### 概要

JPEG の正式な後継規格。**既存の JPEG をロスレスで再圧縮**できる唯一のフォーマット。

#### ツール

- **cjxl** (libjxl に含まれる — cjpegli と同じプロジェクト)
- 既にビルド環境に libjxl が存在するため追加コスト最小

#### 実装

```bash
# ロスレス JPEG → JXL 変換（元のJPEGに完全復元可能）
cjxl input.jpg output.jxl

# Lossy
cjxl input.png output.jxl -q 80
```

#### 優先度: ⭐⭐ （中）

> ブラウザ対応がまだ限定的（Safari のみ）。ただしロスレス JPEG 変換は唯一無二の機能。

## Phase 4: インテリジェント機能

### 4-1. SSIM による自動品質判定

**現状**: ユーザーが品質パラメータを手動設定  
**提案**: SSIM（構造的類似性指標）で「劣化が目立たない限界」を自動検出

#### アルゴリズム

```
1. 品質を高 (95) → 低 (30) の方向で探索
2. 各品質でエンコード → SSIM を計算
3. SSIM > 0.95 (閾値) を満たす最低品質を採用
```

#### ツール

- **ssimulacra2** (libjxl に含まれる — 既にビルド環境に存在)
- Google が開発した次世代の画質評価指標

#### 優先度: ⭐⭐ （中）

> 「品質の数値がわからない」ユーザーにとって非常に有用。ただしエンコードが複数回走るため速度低下に注意。

---

### 4-2. ファイルサイズ目標指定モード

**現状**: 品質パラメータベースの圧縮  
**提案**: 「○MB 以下にしたい」というサイズ目標を指定するモード

#### アルゴリズム

```
1. 目標サイズから逆算して初期品質を推定
2. バイナリサーチで品質を調整
3. 目標サイズ以下かつ最高品質の結果を採用
```

#### UI

```
┌─────────────────────────────┐
│ 圧縮モード:                  │
│ ○ 品質指定 (85%)             │
│ ● サイズ指定 [  7  ] MB 以下  │
└─────────────────────────────┘
```

#### 優先度: ⭐⭐⭐ （高）

> PDF のメール添付やアップロード制限への対応として非常に実用的。


**優先度: ⭐（低）** — ニッチだが業務フロー自動化に有用

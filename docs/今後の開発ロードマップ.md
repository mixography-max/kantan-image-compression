## Phase 1: 圧縮アルゴリズムの高性能化

### 1-1. PNG: Oxipng による前処理の導入

**現状**: `pngquant` → `ECT`（2段階）  
**提案**: `pngquant` → `oxipng` → `ECT`（3段階）

#### 概要

[Oxipng](https://github.com/shssoichiro/oxipng) は Rust 製の PNG 最適化ツール。不要なメタデータチャンクの削除、フィルター戦略の事前選別、インターレース最適化などを行う。

#### 期待効果

- 不要チャンク（tEXt, iCCP, gAMA 等）の除去で **数%** 削減
- ECT への入力を事前に最適化することで、最終出力がわずかに改善

#### 実装方針

```
入力 PNG
  → pngquant (--quality=60-95, 減色)
  → oxipng (-o max --strip all, チャンク削除+フィルター最適化)
  → ECT (-9 --strict, ロスレス再圧縮)
  → 出力 PNG
```

#### 実装手順

1. **ワークフロー追加**: `brew install oxipng`（macOS）、GitHub Releases から取得（Windows）
2. **lib.rs 修正**: `compress_png()` に oxipng ステップを追加
3. **バンドル設定**: `build.yml` で oxipng バイナリをコピー

#### 優先度: ⭐⭐ （中）

> 現状でも十分プロ仕様。得られる改善はごく小さいが、実装コストも低い。

### 1-2. JPEG: Jpegli のパラメータ最適化

**現状**: `cjpegli` で品質パラメータのみ指定  
**提案**: XYB 色空間とプログレッシブスキャンの最適化

#### XYB 色空間の活用

Jpegli の真価は **XYB 色空間**（人間の視覚特性に最適化された色空間）にある。cjpegli はデフォルトで XYB ベースの量子化を使用するが、明示的に確認・設定する。

```bash
# XYB色空間を活用した最適な圧縮
cjpegli input.jpg output.jpg -q 85 --xyb
```

#### プログレッシブスキャン最適化

プログレッシブ JPEG は Web 表示で段階的に画像が表示され、UX が向上する。

```bash
# プログレッシブ + 最適スキャン
cjpegli input.jpg output.jpg -q 85 -p
```

#### 実装手順

1. **lib.rs 修正**: `compress_jpeg()` の cjpegli 引数に `--xyb` と `-p` を追加
2. **設定画面追加**: 「プログレッシブ JPEG」のチェックボックス（オプション）

#### 優先度: ⭐⭐⭐ （高）

> 実装コストがほぼゼロで効果が大きい。最優先で対応すべき。

### 1-3. PDF: 画像抽出→個別最適化→再構築

**現状**: Ghostscript で一括ダウンサンプリング + JPEG 再圧縮  
**提案**: PDF 内の画像を個別に抽出し、Jpegli/ECT で最適化後に PDF に戻す

#### アーキテクチャ

```
入力 PDF
  → pdfimages (画像抽出)
  → 各画像を個別に最適化:
      JPEG → cjpegli で再圧縮
      PNG  → pngquant + ECT
  → img2pdf / Ghostscript で PDF に再構築
  → 出力 PDF
```

#### 実装手順

1. **pdfimages のバンドル**: Poppler ユーティリティに含まれる
2. **画像判定ロジック**: 抽出画像の形式に応じて適切な圧縮器を選択
3. **再構築**: 最適化画像を元の PDF に再埋め込み

#### 優先度: ⭐ （低）

> 実装の複雑度が非常に高い。Ghostscript の一括処理でも十分な結果が得られるため、Phase 1 では見送り推奨。

## Phase 2: 次世代フォーマット対応

### 2-1. WebP 変換対応

#### 概要

Google 開発の画像フォーマット。JPEG/PNG の両方の用途をカバーし、主要ブラウザすべてで対応済み。

#### ツール

- **cwebp** (libwebp に含まれる)
- Homebrew: `brew install webp`
- Windows: [libwebp releases](https://developers.google.com/speed/webp/download)

#### 実装

```bash
# Lossy (JPEG代替)
cwebp -q 80 input.jpg -o output.webp

# Lossless (PNG代替)
cwebp -lossless input.png -o output.webp
```

#### 優先度: ⭐⭐⭐ （高）

> Web 制作者にとっての需要が非常に高い。実装も容易。

---

### 2-2. AVIF 変換対応

#### 概要

AV1 ベースの画像フォーマット。**現時点で Web 利用可能な形式の中で最高の圧縮率**を誇る。

#### ツール

- **avifenc** (libavif に含まれる)
- Homebrew: `brew install libavif`
- Windows: [libavif releases](https://github.com/AOMediaCodec/libavif/releases)

#### 実装

```bash
avifenc --min 20 --max 30 -s 6 input.png output.avif
```

#### 優先度: ⭐⭐ （中）

> 圧縮率は最強だが、エンコード速度が遅い。設定 UI のバランスが課題。

---

### 2-3. JPEG XL 変換対応

#### 概要

JPEG の正式な後継規格。**既存の JPEG をロスレスで再圧縮**できる唯一のフォーマット。

#### ツール

- **cjxl** (libjxl に含まれる — cjpegli と同じプロジェクト)
- 既にビルド環境に libjxl が存在するため追加コスト最小

#### 実装

```bash
# ロスレス JPEG → JXL 変換（元のJPEGに完全復元可能）
cjxl input.jpg output.jxl

# Lossy
cjxl input.png output.jxl -q 80
```

#### 優先度: ⭐⭐ （中）

> ブラウザ対応がまだ限定的（Safari のみ）。ただしロスレス JPEG 変換は唯一無二の機能。

## Phase 3: UI/UX の高度化

### 3-1. 並列処理（マルチコア活用）

**現状**: ファイルを1つずつ順番に圧縮  
**提案**: CPU コア数に応じた並列圧縮

#### 実装方針

Rust の **Rayon** クレートを使用し、スレッドプールで並列実行。

```rust
use rayon::prelude::*;

files.par_iter().for_each(|file| {
    compress_file(file, &settings);
});
```

#### Cargo.toml 追加

```toml
rayon = "1"
```

#### UI 側対応

- プログレスバーを「全体の進捗」と「個別ファイルの進捗」に分離
- 「並列数」設定（自動 / 手動指定）

#### 優先度: ⭐⭐⭐ （高）

> 大量ファイル処理時のパフォーマンスが劇的に向上。Rayon の導入は非常に容易。

---

### 3-2. 圧縮前後のプレビュー比較機能

**現状**: 圧縮結果はファイルサイズのみ表示  
**提案**: 圧縮前後の画像をスライダーで並べて比較

#### 実装方針

- React コンポーネントで「ビフォー/アフター」スライダーを実装
- 圧縮後の画像を一時的に Base64 エンコードしてフロントエンドに送信
- ピンチズーム対応で細部の確認も可能に

#### UI イメージ

```
┌─────────────────────────────┐
│  Before  ◄━━━|━━━►  After  │
│          スライダー          │
│  [元: 2.4 MB]  [圧縮: 480 KB]│
│         削減率: 80%          │
└─────────────────────────────┘
```

#### 優先度: ⭐⭐ （中）

> ユーザー体験を大きく向上させるが、実装コストはやや高い。

---

### 3-3. ドラッグ＆ドロップでフォルダ一括圧縮

**現状**: ファイル単位のドラッグ＆ドロップ  
**提案**: フォルダをドロップすると中身を再帰的に圧縮

#### 実装手順

1. ドロップされたパスがディレクトリか判定
2. 再帰的にファイルを列挙し、対応形式のみフィルタ
3. 元のフォルダ構造を保持して出力

#### 優先度: ⭐⭐⭐ （高）

> 実装コストが低く、実用性が非常に高い。

## Phase 4: インテリジェント機能

### 4-1. SSIM による自動品質判定

**現状**: ユーザーが品質パラメータを手動設定  
**提案**: SSIM（構造的類似性指標）で「劣化が目立たない限界」を自動検出

#### アルゴリズム

```
1. 品質を高 (95) → 低 (30) の方向で探索
2. 各品質でエンコード → SSIM を計算
3. SSIM > 0.95 (閾値) を満たす最低品質を採用
```

#### ツール

- **ssimulacra2** (libjxl に含まれる — 既にビルド環境に存在)
- Google が開発した次世代の画質評価指標

#### 優先度: ⭐⭐ （中）

> 「品質の数値がわからない」ユーザーにとって非常に有用。ただしエンコードが複数回走るため速度低下に注意。

---

### 4-2. ファイルサイズ目標指定モード

**現状**: 品質パラメータベースの圧縮  
**提案**: 「○MB 以下にしたい」というサイズ目標を指定するモード

#### アルゴリズム

```
1. 目標サイズから逆算して初期品質を推定
2. バイナリサーチで品質を調整
3. 目標サイズ以下かつ最高品質の結果を採用
```

#### UI

```
┌─────────────────────────────┐
│ 圧縮モード:                  │
│ ○ 品質指定 (85%)             │
│ ● サイズ指定 [  7  ] MB 以下  │
└─────────────────────────────┘
```

#### 優先度: ⭐⭐⭐ （高）

> PDF のメール添付やアップロード制限への対応として非常に実用的。

### B. 圧縮履歴の保存・エクスポート

- 圧縮した全ファイルの履歴（元サイズ、圧縮後サイズ、日時）を保存
- CSV エクスポート機能で報告書作成に活用
- 「今月の総削減量」等の統計表示

**優先度: ⭐⭐（中）** — 業務利用で重宝される

Tauri アプリとは別に、CI/CD やバッチ処理で使えるコマンドラインインターフェースを提供。

```bash
hamham-compress --quality 85 --format jpeg --output ./compressed/ ./images/*.jpg
```

**優先度: ⭐⭐（中）** — 開発者・パワーユーザー向け

---

### D. 監視フォルダ（ホットフォルダ）機能

指定フォルダに画像が追加されると自動で圧縮する常駐モード。

```
監視: ~/Desktop/圧縮待ち/
出力: ~/Desktop/圧縮済み/
→ ファイルが追加されると自動で圧縮開始
```

**優先度: ⭐（低）** — ニッチだが業務フロー自動化に有用

---

### E. 画像リサイズ機能

圧縮と同時にリサイズ（最大幅・最大高さの指定）を行う機能。

- 「長辺を 1920px 以下に」等の指定
- Web 掲載用に最適なサイズに自動調整
- アスペクト比は維持

**優先度: ⭐⭐⭐（高）** — 圧縮と並んで最もリクエストされる機能
